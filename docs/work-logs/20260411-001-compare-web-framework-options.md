# 작업 기록

- 일시: 2026-04-11
- 작업자: Codex
- 작업 내용: 영어 암기 노트 웹 서비스 구현 전, 사용할 웹 프레임워크 후보를 비교하고 현재 요구사항에 맞는 추천안을 정리함

## 배경

현재 프로젝트는 다음 요구사항을 가진다.

- 웹페이지에서 여러 개의 `jpg`, `png` 이미지 파일을 업로드할 수 있어야 함
- 업로드된 이미지에서 영어 단어와 뜻을 추출해야 함
- 추출 결과를 단어장 형태로 웹에서 보여줘야 함
- 결과는 프린터로 출력할 수 있어야 함
- 이후 OCR 및 출력 레이아웃 구현이 필요함

이 요구사항을 기준으로 프레임워크 후보를 비교했다.

## 비교 기준

- 여러 파일 업로드 구현 편의성
- 서버 처리 및 OCR 연동 편의성
- 결과 페이지 렌더링 편의성
- 인쇄 친화적인 웹페이지 구현 적합성
- 초기 개발 속도
- 추후 확장성

## 후보 비교

### 1. Next.js

- 장점:
  React 기반이라 화면 구성과 상태 관리가 익숙한 생태계를 활용하기 좋다.
- 장점:
  프론트엔드 페이지와 서버 API를 한 프로젝트 안에서 함께 관리하기 쉽다.
- 장점:
  업로드 화면, 결과 화면, 인쇄용 화면을 하나의 웹앱으로 묶기 좋다.
- 장점:
  HTML/CSS 기반 인쇄 레이아웃 구현에 유리하다.
- 장점:
  추후 인증, 저장 기능, 배포 확장도 비교적 수월하다.
- 단점:
  OCR 처리 자체가 무거워지면 서버 구성 분리가 필요할 수 있다.
- 단점:
  이미지 처리 파이프라인을 Node.js에서만 모두 해결하려 하면 라이브러리 선택 폭이 제한될 수 있다.

### 2. Next.js + FastAPI

- 장점:
  프론트엔드는 Next.js로, OCR 및 파일 처리는 FastAPI로 분리할 수 있어 역할이 명확하다.
- 장점:
  Python 생태계의 OCR 및 이미지 처리 도구를 활용하기 쉽다.
- 장점:
  복잡한 텍스트 추출이나 후처리 로직이 커질수록 구조적으로 유리하다.
- 단점:
  초기 구조가 프론트엔드와 백엔드로 나뉘어 설정이 늘어난다.
- 단점:
  로컬 개발과 배포 시 관리 포인트가 하나 더 생긴다.

### 3. Nuxt

- 장점:
  Vue 기반 풀스택 개발을 선호한다면 생산성이 좋다.
- 장점:
  서버 API와 페이지를 함께 구성하기 편하다.
- 장점:
  화면 렌더링과 인쇄용 페이지 구성도 충분히 가능하다.
- 단점:
  현재 비교 기준에서는 Next.js보다 팀이나 자료 측면에서 선택 근거가 상대적으로 약하다.
- 단점:
  OCR 연동 자체의 이점은 Next.js 대비 특별히 더 크지 않다.

### 4. React Router

- 장점:
  React 기반으로 유연하게 구성할 수 있다.
- 장점:
  파일 업로드 처리 패턴도 공식 가이드가 존재한다.
- 단점:
  현재 요구사항에서는 Next.js보다 초기 구조 선택이 조금 더 직접적이고 수동적이다.
- 단점:
  업로드, 서버 처리, 결과 출력까지 빠르게 한 번에 묶어 시작하는 관점에서는 Next.js보다 덜 간편하다.

### 5. Django

- 장점:
  관리자 기능, 서버 렌더링, 파일 업로드 등 전통적인 웹 기능이 안정적이다.
- 장점:
  Python 기반이라 OCR 연동이 편하다.
- 단점:
  프론트엔드 인터랙션과 현대적인 UI 구성은 별도 선택이 더 필요할 수 있다.
- 단점:
  현재 프로젝트처럼 업로드 후 즉시 결과 화면과 인쇄 화면을 세련되게 제공하려면 프론트 구성이 다소 무거워질 수 있다.

## 현재 기준 추천

### 추천 1: Next.js

현재 요구사항만 보면 가장 균형이 좋은 선택이다.

- 업로드 UI, 결과 페이지, 인쇄 레이아웃을 빠르게 시작하기 좋다.
- 초기 코드 구조를 단일 프로젝트로 간단하게 만들 수 있다.
- MVP를 만드는 속도가 빠를 가능성이 높다.

### 추천 2: Next.js + FastAPI

OCR과 이미지 처리 로직이 중요하고 복잡해질 가능성이 높다면 이 조합이 가장 실용적이다.

- 웹 UI는 Next.js가 담당
- OCR 및 후처리는 FastAPI가 담당
- 기능이 커져도 구조를 유지하기 좋음

## 현재 판단

초기 시작은 `Next.js + TypeScript`가 가장 적합하다.

이유는 다음과 같다.

- 웹 서비스 형태를 빠르게 만들 수 있음
- 파일 업로드와 결과 표시 흐름을 바로 구현하기 좋음
- 프린터 출력용 레이아웃을 웹 기술로 자연스럽게 다룰 수 있음
- 필요할 경우 나중에 OCR 전용 백엔드를 분리하는 확장도 가능함

## 결정 사항

- 1차 추천 프레임워크: `Next.js`
- OCR 복잡도가 커질 경우 고려할 확장안: `Next.js + FastAPI`
