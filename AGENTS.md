# AGENTS.md

이 문서는 `english_memory_note_maker`에서 작업하는 코딩 에이전트가 따라야 하는 프로젝트 전용 규칙을 정의한다.  
앞으로 이 저장소에서 기능 추가, 수정, 문서 갱신을 수행할 때는 이 문서의 규칙을 기본 기준으로 삼는다.

## 1. 제품 목표

- 이 프로젝트는 영어 단어가 포함된 이미지에서 `단어 + 뜻`을 추출하고, 학습용 암기 노트를 생성하는 웹 앱이다.
- 최종 산출물의 중심은 브라우저 인쇄가 아니라 `PDF`와 `XLS` 다운로드다.
- 사용자는 결과 화면에서 미리보기를 확인한 뒤, `A4` 인쇄를 고려한 문서를 내려받는다.

## 2. 절대 유지해야 하는 현재 제품 규칙

- 브라우저 인쇄 전용 페이지 기능은 사용하지 않는다.
- `/print` 같은 별도 인쇄 페이지를 다시 만들지 않는다.
- 다운로드 중심 흐름을 유지한다.
  - `PDF` 다운로드
  - `XLS` 다운로드
- 문제지 번호는 셔플 순번이 아니라 `원본 추출 순서 번호`를 사용한다.
- 문서 마지막에는 반드시 `정답지`가 포함되어야 한다.
  - PDF: 마지막 페이지들에 정답지 표 포함
  - XLS: 마지막 시트에 정답지 포함
- 문제지는 다음 두 학습 유형을 지원해야 한다.
  - `단어를 보고 뜻 쓰기`
  - `뜻을 보고 단어 쓰기`
- 각 학습 유형은 `문항 수`가 아니라 `반복 회차` 기준으로 생성한다.
  - `1`이면 전체 세트 1회
  - `2`이면 전체 세트 2회
  - `3`이면 전체 세트 3회
- 반복 회차는 `0`도 허용한다.
  - `0`이면 해당 유형은 결과물에서 완전히 제외한다.

## 3. 출력물 규칙

### PDF

- `A4` 기준으로 생성한다.
- 흑백 인쇄를 기본으로 한다.
- 배경은 흰색이어야 한다.
- 장식성 색상 사용을 피한다.
- 문제지는 표 형태여야 한다.
- 개별 카드형 박스 UI로 되돌리지 않는다.
- 표 헤더는 매 페이지에 보여야 한다.
- 문제지 표 구조는 아래를 유지한다.
  - `번호 | 문항 | 답안`
- 오른쪽 `답안` 칸은 비워 둔다.
- 각 행에 `뜻`, `단어`, 밑줄 라벨을 다시 넣지 않는다.
- `번호`와 `문항` 사이에는 시각적 칸막이가 있어야 한다.

### XLS

- PDF와 동일한 정보 구조를 반영한다.
- 문제 시트는 `번호 | 문항 | 답안` 구조를 사용한다.
- `답안` 칸은 빈 상태여야 한다.
- 마지막 시트는 정답지여야 한다.
  - `번호 | 단어 | 뜻`
- 인쇄 시 `A4` 기준으로 최대한 맞춰지도록 페이지 설정을 포함한다.
- 헤더 행은 인쇄 2페이지 이후에도 반복되어야 한다.

## 4. 미리보기와 다운로드의 일치성

- 화면 미리보기와 실제 `PDF/XLS` 결과는 가능한 한 같은 구조를 따라야 한다.
- 사용자가 화면에서 설정한 반복 회차와 섹션 구성은 다운로드 파일에도 그대로 반영되어야 한다.
- 미리보기와 다운로드가 서로 다른 셔플 결과를 갖지 않도록 주의한다.

## 5. 데이터 규칙

- `samples/expected/result.json`의 항목 순서는 원본 번호 기준으로 사용된다.
- 정답지 번호는 항상 이 원본 순서를 따라야 한다.
- 문제지에서 셔플이 있더라도 표시 번호는 원본 번호를 유지해야 한다.

## 6. UI/UX 규칙

- 사용자가 이해하기 어려운 기술적 옵션보다 학습 흐름 중심 용어를 우선한다.
- `문항 수`처럼 오해를 부르는 표현보다 실제 동작을 설명하는 표현을 사용한다.
  - 예: `반복 회차`
- 설정 UI를 추가할 때는 아래를 우선 고려한다.
  - 사용자가 결과를 바로 예측할 수 있는가
  - 미리보기와 다운로드 결과가 일치하는가
  - 원본 번호와 정답지 대조가 쉬운가

## 7. 코드 변경 규칙

- 출력물 관련 로직을 바꿀 때는 아래 파일들을 함께 확인한다.
  - `app/page.tsx`
  - `app/globals.css`
  - `lib/memory-note.ts`
  - `lib/memory-note-export.ts`
  - `lib/types.ts`
- PDF와 XLS 중 하나만 바꾸지 말고, 둘 다 같은 제품 규칙을 만족하는지 확인한다.
- 번호 체계, 정답지, 반복 회차 관련 로직은 한쪽만 수정하지 않는다.
- 출력 포맷의 구조를 바꿨다면 미리보기도 같이 검토한다.

## 8. 문서 업데이트 규칙

- 사용자에게 보이는 기능이 바뀌면 문서도 함께 갱신한다.
- 최소 업데이트 대상:
  - `README.md`
  - `docs/PROJECT_SPEC.md`
  - `docs/DESIGN.md`
- 의미 있는 작업 단위가 끝나면 `docs/work-logs/`에 작업 기록을 추가한다.
- 샘플 데이터 구조나 의미가 바뀌면 `samples/README.md`도 갱신한다.

## 9. 검증 규칙

- 코드 변경 후 가능하면 `npm run build`로 검증한다.
- 출력 로직을 바꾼 경우 아래를 우선 확인한다.
  - PDF 생성 성공 여부
  - XLS 생성 성공 여부
  - 헤더 반복 여부
  - 원본 번호와 정답지 일치 여부
  - 반복 회차 설정 반영 여부

## 10. 금지 사항

- 브라우저 인쇄 중심 기능으로 다시 회귀하지 않는다.
- 별도 `/print` 페이지를 부활시키지 않는다.
- 색상 위주의 인쇄 레이아웃으로 바꾸지 않는다.
- 문제 번호를 셔플 순번으로 되돌리지 않는다.
- 정답지를 제거하지 않는다.
- 반복 회차를 단순 문항 수 제한 기능으로 바꾸지 않는다.

## 11. 작업 시 우선순위

변경안을 판단할 때는 아래 우선순위를 따른다.

1. 원본 번호와 정답지 대조 가능성
2. PDF/XLS 출력물의 실사용성
3. 미리보기와 다운로드 결과의 일치성
4. 흑백 A4 인쇄 적합성
5. 사용자의 설정 이해 용이성
6. 코드 구조의 일관성

## 12. 에이전트 행동 원칙

- 요구가 모호할 때는 현재 제품 규칙을 깨지 않는 방향으로 해석한다.
- 새 기능이 기존 출력 규칙과 충돌하면, 무리하게 구현하지 말고 충돌 지점을 먼저 드러낸다.
- 사용자가 기능 변경을 요청했을 때는 `PDF`, `XLS`, 미리보기, 문서까지 함께 영향을 검토한다.
- 이 문서와 실제 구현이 어긋나면, 코드를 바꿨다면 문서도 함께 맞춘다.
